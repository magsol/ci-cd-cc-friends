We've all been here. What changed? Why? Is this a breaking change?

Everyone recognizes these messages. The point: unstructured messages mean you can't automate anything — no changelogs, no auto-versioning, no way to quickly understand impact.

Conventional Commits

<type>(scope): <description>

[optional body]

[optional footer(s)]

conventionalcommits.org

The spec is simple: type, optional scope, description. Body and footers are optional but useful for BREAKING CHANGE notes.

Common Types

  • feat — new feature
  • fix — bug fix
  • docs — documentation
  • style — formatting
  • refactor — restructure
  • test — add/fix tests
  • ci — CI/CD changes
  • chore — maintenance

BREAKING CHANGE: in the footer, or ! after the type

feat and fix are the most important because they map directly to SemVer: feat = minor bump, fix = patch bump, BREAKING CHANGE = major.

Scopes in Practice: pmbench

14 development threads = 14 CC scopes

Each thread maps to a scope in our commit messages

In pmbench, we defined 14 "development threads" — focus areas like infrastructure, experiment management, visualization, etc. Each one becomes a CC scope. This means anyone can glance at a commit and immediately know which area of the codebase is affected. These are enforced via commitlint's scope-enum rule.

Real Commits from pmbench

These are real commits from the past week. Point out: - fix(infra) — a bug fix in the infrastructure thread - chore(config) — maintenance work on configuration, collaborating with @kadams - refactor(config) — restructuring, same scope but different type - chore(experiment) and fix(experiment) — linting cleanup vs actual bug fix, both in the experiment management thread. The types tell you the nature of the change, the scopes tell you where.

This is the key insight. CC isn't just about readable messages — it's machine-readable metadata that powers your entire release pipeline.

commitlint

Validates commit messages against CC rules


// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'scope-enum': [2, 'always', [
      'collab', 'compare', 'config', 'docs',
      'experiment', 'explain', 'framework',
      'hyperparam', 'infra', 'obs', 'prov',
      'search', 'server', 'viz',
    ]],
  },
};

Enforced locally via husky git hooks

commitlint is the gatekeeper. Combined with husky, it runs on every commit locally so bad messages never even reach the remote. Mention: scope-enum is optional but helpful for consistency.

semantic-release

Reads CC history → determines next version → publishes

Commit

  • fix(...)
  • feat(...)
  • feat(...)!

Version Bump

  • 1.0.1 (patch)
  • 1.1.0 (minor)
  • 2.0.0 (major)

Also creates git tags, GitHub/GitLab releases, and publishes packages

semantic-release is the crown jewel. No more debating version numbers. The commit history IS the release plan. Mention it supports both GitHub and GitLab.

Auto-generated CHANGELOG


## [1.2.0] - 2026-03-13

### Features
- **pipeline:** add batch processing for phenotype extraction (a1b2c3d)

### Bug Fixes
- **io:** handle missing columns in input CSV gracefully (e4f5g6h)

### BREAKING CHANGES
- **models:** restructure model registry API (i7j8k9l)

Zero manual effort. Generated from commit history.

This is what your users/team actually sees. Fully automated, grouped by type, with links back to commits. No more manually writing release notes.

GitHub Actions


# .github/workflows/release.yml
name: Release
on:
  push:
    branches: [main]
jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
This is what we'll use in the demo. Push to main, semantic-release analyzes commits since last tag, bumps version, generates changelog, creates a GitHub release. All automatic.

GitLab CI


# .gitlab-ci.yml
release:
  stage: deploy
  image: node:20
  only:
    - main
  script:
    - npm ci
    - npx semantic-release
  variables:
    GITLAB_TOKEN: $GITLAB_TOKEN

Same tooling, just swap the CI config and token variable.
semantic-release has a @semantic-release/gitlab plugin.

For our internal work, we use GitLab. The good news: semantic-release supports GitLab natively with the gitlab plugin. The config is nearly identical. Mention @semantic-release/gitlab replaces @semantic-release/github.
